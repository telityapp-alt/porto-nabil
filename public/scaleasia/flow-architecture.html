Platform Architecture
Content-first design with search, filtering, and operator profiles
Design Philosophy
Discoverability Over Engagement Loops
ScaleAsia is not a social feed. It is a knowledge base optimized for search and filtering. Operators arrive with specific questions ("How do I handle transfer pricing for Indonesia PT PMA?") and need fast path to relevant playbook. Architecture prioritizes SEO, taxonomy clarity, and filter precision over infinite scroll and recommendation algorithms.
Core Principles
  • Search-first: Every page indexed for organic discovery. Long-tail keywords ("Indonesia PT PMA beneficial ownership disclosure") map to specific playbooks.
  • Taxonomy-driven navigation: Browse by operating area (Entity, Compliance, Hiring, Tax, Banking, Legal, Operations) or by jurisdiction.
  • Progressive disclosure: Playbook previews show context and decision tree. Full content behind single click, no multi-page pagination.
  • Static content, dynamic filtering: Playbooks are static markdown, filters applied client-side for instant results.
Frontend Stack
Static Site with Client-Side Search
Same architecture as HR Asia: Next.js static export for content pages, React islands for interactive features (search, filtering, workshop registration). Majority of traffic lands on playbook pages from Google — those pages ship zero client-side JS for instant load. Search and filtering hydrate on-demand.
Technology Choices
Framework
Next.js 14 (App Router) with static export. RSC for operator profiles and workshop booking.
Content
MDX for playbooks with frontmatter (operating_area, jurisdictions, author, publish_date, tags).
Search
Pagefind (static search index) or Algolia DocSearch for instant client-side search across playbooks.
Styling
Tailwind CSS with custom design system (same retro palette as HR Asia for brand consistency).
Typography
Source Sans 3 (body) + Lora (headings) for professional, readable long-form content.
Backend & Data
Minimal Backend, Git-Based CMS
Backend operations limited to: workshop registration, operator profile management, and weekly regulatory update emails. Content authoring happens in Git (operators submit playbooks via PR, editorial review before merge). No database for content — markdown files are source of truth. Database only for user accounts, workshop registrations, and email subscriptions.
Backend Architecture
Functions
Cloudflare Workers for workshop booking, email subscriptions, operator profile CRUD.
Database
Cloudflare D1 (SQLite at edge) for user accounts, workshop enrollments, email list.
CMS
Git-based (GitHub repo). Operators submit playbooks via PR (markdown + frontmatter). Editorial review, then merge to main triggers deploy.
Email
Resend API for weekly regulatory update newsletter and workshop confirmation emails.
Auth
Clerk or Auth.js for operator accounts (GitHub OAuth preferred for operator identity verification).
Content Pipeline
Operator Contribution → Editorial Review → Deploy
Operators draft playbooks in markdown (local editor or GitHub web UI). Submit PR with frontmatter (operating area, jurisdictions, tags). Editorial team reviews for: (1) Structure completeness (context, decision tree, execution steps, costs, pitfalls), (2) Operator credibility (profile verification), (3) Uniqueness (does this add value vs existing playbooks?). Approved PRs merge to main, trigger static build and deploy to edge within 5 minutes.
Contribution Workflow
  • Step 1 — Draft: Operator writes playbook in markdown following template (frontmatter + sections: context, decision tree, steps, costs, pitfalls).
  • Step 2 — Submit: Open GitHub PR. Automated checks validate frontmatter schema and markdown linting.
  • Step 3 — Review: Editorial team reviews structure, verifies operator credentials, checks for duplication with existing content.
  • Step 4 — Publish: PR approved and merged. CI/CD builds static site, deploys to Cloudflare Pages, playbook live within 5 minutes.
  • Step 5 — Promote: New playbook featured in weekly regulatory update email. Operator profile updated with contribution credit.
Search & Filtering
Multi-Dimensional Playbook Discovery
Operators filter playbooks by: (1) Operating area (Entity, Compliance, Hiring, Tax, Banking, Legal, Operations), (2) Jurisdiction (Singapore, Indonesia, Philippines, India, Malaysia, Japan), (3) Author (find all playbooks by specific operator), (4) Tags (transfer pricing, EOR, AGM, bank account, beneficial ownership). Filters combine with AND logic (show me Compliance playbooks for Indonesia about beneficial ownership). Search works across playbook title, summary, and body text.
Implementation
  • Static index: Playbook metadata extracted at build time into JSON index file (title, summary, frontmatter, URL).
  • Client-side filtering: React component loads index, applies filters instantly without server round-trip.
  • Full-text search: Pagefind generates static search index at build time. Client-side JS hydrates search UI on-demand.
  • SEO optimization: Each playbook is standalone HTML page with semantic structure (h1, h2, h3 hierarchy, schema.org HowTo markup).
Operator Profiles
Credibility & Contribution History
Every operator has public profile showing: (1) Entities operated (jurisdictions, entity types, years active), (2) Contribution history (playbooks authored, workshops facilitated), (3) Verified credentials (LinkedIn, company website, GitHub verified). Profile builds operator reputation and helps users assess playbook credibility ("This operator ran 5 entities across SEA for 3 years — their transfer pricing playbook is field-tested").
Profile Data Model
Identity
Name, photo, bio, LinkedIn URL, company website (verified via OAuth or manual review).
Experience
List of entities operated (jurisdiction, entity type, start year, end year or "present").
Contributions
Playbooks authored (with publish date and view count), workshops facilitated (with participant count).
Verification
Badge for verified operators (manual review confirms multi-entity experience is real).
Workshop Platform
Live Sessions with Registration & Recording
Workshops run live via Zoom (or similar). Registration handled via ScaleAsia platform (workshop detail page → register button → collect email + entity info). Post-workshop, recording and slide deck uploaded to platform and linked from workshop page. Attendees get email with recording link. Non-attendees can expense workshop (corporate reimbursement flow) to access recording and materials.
Workshop Workflow
  • Pre-workshop: Operator submits workshop proposal (topic, outline, date/time). Editorial approval. Workshop page created with registration form.
  • Registration: User fills form (name, email, entities operated, specific questions for operator). Email confirmation sent with Zoom link.
  • Live session: Operator facilitates 90-120 min workshop. Recording captured automatically.
  • Post-workshop: Recording uploaded to Cloudflare Stream (or YouTube unlisted). Slide deck attached as PDF. Workshop page updated with recording embed.
  • Access control: Attendees get free access. Non-attendees can expense ("Expense a program" flow submits receipt to employer for reimbursement).
Architecture Trade-offs
Why static-first: Playbooks are evergreen content that changes infrequently (quarterly updates to reflect regulatory changes). Pre-rendering ensures instant load for SEO-driven traffic. Trade-off: real-time features limited (no live chat, no instant notifications). We accept this — async community (Slack) handles real-time operator discussions. Why Git-based CMS: Operators are technical (CFOs, finance leads, legal ops). They are comfortable with markdown and PR workflows. Git provides version control, attribution, and editorial review built-in. Trade-off: non-technical operators face steeper contribution barrier. We accept this — target audience is operator who can navigate GitHub, not general public. Why minimal backend: Focus engineering effort on content quality and search UX, not building CMS features. Trade-off: no rich text editor, no in-app collaboration. We accept this — markdown simplicity keeps content portable and future-proof.
